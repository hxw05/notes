# cmd

本分类下的笔记，主要记录在日常学习和工作中经常用到的一些命令行的pattern，以便将来查阅和复用。

## GUI与CLI

我接触到的第一门“编程语言”（实际上是脚本语言）是Windows下的批处理文件（GitHub上面称为Batchfile），它相当于cmd中指令的集合；这门脚本语言塑造了我对过程式编程的基本认识。

命令行是计算机的一个自然、原始、具有丰富的历史同时在当今仍然强大的接口，一些我们日常使用的GUI工具（不是全部）可以在命令行中找到其等价，例如编辑器（vim/emacs/nano，其中vim奠定了一种标准易用的命令行buffer操作模式）、Markdown阅读器（glow）、Postman（实际上就是HTTP客户端，命令行替代品有最基本的cURL，以及hurl等）等。事实上，在没有引入图形化操作系统API之前，使用高级程序设计语言构建出的工具均以命令行（stdin、stdout）为默认接口；命令行中存在的许多概念均与经典的计算机操作系统知识、基础的编程语言知识息息相关。

但由于命令行的原始性，在一些角度来看，其使用并不直观，加之学习曲线陡峭，在很多方面都*显得*不如图形界面（GUI）易用、高效。如果一个程序只提供了CLI版本，那么它所面向的大概率只是开发者或资深用户，而几乎与最庞大的大众群体绝缘（除非实在是太过于好用，以至于所有人都要顺着搜索引擎或社交媒体过来体验）。

如今，随着计算机技术的发展，我们早已脱离了GUI需求的初级阶段，转向审美层面的需求；许多“无法在命令行中实现的功能”都是在GUI普及的背景下诞生并不断发展，例如浏览器及其背后的网页标准。GUI是当今主流的计算机交互接口。

但命令行在一些场景下仍然有着不可替代的作用。例如coding agent、非交互式（non-interactive）任务（脚本）、运维管理等。这些场景下，GUI成为了某种程度上的瓶颈所在——可能是操作效率上的，也可能是资源占用率上的。这并不是说GUI不能存在于或用于这类场景中，只是不再适合。

命令行还可被视作一种系统自带的“脚本语言”。在不安装任何其它语言的解释器之前，我们同样可以借助shell脚本（Windows下为ps脚本、bat脚本）等构建一套执行流程，并在系统中原生执行。

总而言之，GUI相比CLI，更像是专为广大用户（包含专业计算机用户、程序设计者在内）设计的通用接口。CLI与GUI不是替代关系，而是相辅相成的两种交互接口，有其各自的侧重面。当今操作系统中，CLI通常以一个“终端”的GUI应用的方式暴露。

All in 命令行几乎需要经过长期的使用或专门的训练才能达到，并且学习的投入产出比并不会如想象中的那样高。虽然几乎每一个命令行工具都可以通过`man cmd`打开其手册（manual），其中完整、详尽地记录了这一指令的用法，但我觉得通过查阅这类手册去刻意学习困难重重，因为一些命令行工具对于普通用户而言存在相当大的leaky abstraction，有许多参数的含义在真正遇到相关场景之前是难以解释的。所以我并不指望自己能够达到一个“对命令行工具的每一个参数都了如指掌”的程度，而大概只需要是“对自己需要用到的每一个命令行工具的相关参数了如指掌”。

总之，本着体验过部分命令行操作，认为命令行确实能在某些方面提高操作效率的朴素认知，我倾向于将命令行与图形化界面结合使用，这尤其适用于macOS这种*nix类系统——它们都有着一套高度标准化的命令行指令（等）体系，使得你可以在几乎所有类似系统上都有着相似（或者说“神似”，但不相同，例如GNU vs. BSD）的常用指令体验，这种体系已是当今主流开发体验的de facto指令体系。
