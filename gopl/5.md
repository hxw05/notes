# Ch5. Functions

## 函数定义

使用以下格式来定义一个函数
```
func name(param-list) (result-list) {
    body
}
```

在形参列表中，相同类型的参数的类型可以提取出来：`x int, y int`→`x, y int`。

函数也是一种类型。一个函数的类型又被称为这个函数的**签名（signature）**。当两个函数的形参列表和返回值列表分别相同（顺序、类型相同）时，认为这两个函数具有相同的类型，该比较不受参数名、返回值名以及是否使用提取了公共部分的写法（factored form）影响。

Go中不支持默认参数，同时也没有按名指定参数的方式（类似于Python的那种）。每一个函数调用都必须按顺序提供完整的参数列表。

函数调用是值传递的（passed by value），函数内部总是获取到的是调用者传入的参数的拷贝，对拷贝做修改不会影响到原有的值，除非其数据类型包含对数据的间接访问（如指针、切片、map、函数、通道）。

如果只有函数的定义而没有函数体，就说明这个函数是在其它语言中实现的，例如`math.Sin`。

```go
func Sin(x float64) float64 // implemented in assembly language
```

## 递归

在Ch2的[作用域](./2.md#作用域-scope)一节提到过，在package level定义的标识符不受顺序影响，此机制使得函数的递归成为可能。下面是一个递归函数的例子，该函数用于打印出解析的HTML节点的树状结构。

```go
func main() {
    doc, err := html.Parse(os.Stdin)
    if err != nil {
        fmt.Fprintf(os.Stderr, "outline: %v\n", err)
        os.Exit(1)
    }
    outline(nil, doc)
}
func outline(stack []string, n *html.Node) {
    if n.Type == html.ElementNode {
        stack = append(stack, n.Data) // push tag
        fmt.Println(stack)
    }
    for c := n.FirstChild; c != nil; c = c.NextSibling {
        outline(stack, c)
    }
}
```

特别注意：这里的stack只有进栈而没有出栈。这使得更深一层的调用中对stack的追加虽然会改变其底层数组的内容，但是不影响上一层函数中的那个stack上的`ptr`和`len`（将切片想象成结构体更好理解），因为stack是值传递。于是在更深一层的函数对底层数组的修改并不会影响上一层函数所能看到的底层数组中的内容，也就不影响回溯以后进入到下一个递归路径时，stack的起点位置。换句话说，值传递的特性在这里代替了出栈操作。

作为对比，JS与此不同，由于全程用作栈的数组是作为引用传递的，回溯以后需要显式地出栈才能让内容正常输出。

以上程序的输出示例如下，可以看到在输出了`[html head meta]`之后又正常输出了`[html head title]`。程序中这里发生的是原本的`meta`叶子被输出后，由于没有更深层的元素发生回溯，回溯之后`meta`并没有被删除，而是直接在输出`title`的那个分支中被`append`覆盖为了`title`（作为垃圾数据），其它的以此类推。
```
[html]
[html head]
[html head meta]
[html head title]
[html head link]
[html body]
[html body div]
[html body div]
[html body div div]
[html body div div form]
[html body div div form div]
[html body div div form div a]
```

> As you can see by experimenting with outline, most HTML documents can be processed with only a few levels of recursion, but it’s not hard to construct **pathological** web pages that require extremely deep recursion. (p124)

通过递归输出网页的结构是一种很自然的方法。但是网页并不总是这么简单，尤其是在现有的前端工具链中，在复杂的业务逻辑下产生一个嵌套非常深的网页并不难。如果仍然使用递归来遍历，我们自然会考虑到是否会产生调用栈溢出（stack overflow）。在相当一部分编程语言中，这是成立的，因为它们使用的是固定的栈大小（常见64KB到2MB）；Go中的调用栈大小是动态的，可以根据需要扩展到至多GB级别，所以在Go中使用递归来处理问题的时候大概率不会遇到栈溢出（但仍然可能遇到其它递归问题）。

## 多个返回值

Go中的函数可以返回多个返回值，最常见的是连同结果一起返回一个错误或者布尔值。

```go
func findLinks(url string) ([]string, error) {
    resp, err := http.Get(url)
    if err != nil {
        return nil, err
    }
    if resp.StatusCode != http.StatusOK {
        resp.Body.Close()
        return nil, fmt.Errorf("getting %s: %s", url, resp.Status)
    }
    doc, err := html.Parse(resp.Body)
    resp.Body.Close()
    if err != nil {
        return nil, fmt.Errorf("parsing %s as HTML: %v", url, err)
    }
    return visit(nil, doc), nil
}
```

Note: 上面的代码演示了无论在正常还是异常情况下，都需要对closable资源进行显式close。Go虽然有GC机制，但是你不能指望它会回收掉那些还处于开启状态的资源。

> Go’s garbage collector recycles unused memory, but do not assume it will release unused operating system resources like open files and network connections. They should be closed explicitly. (p125)

一个返回多个值的函数可以直接作为一个接收多个参数函数的参数。这种写法在生产环境下用的很少，但是很适合用来传入调试输出。

```go
log.Println(findLinks(url))
// 相当于
links, err := findLinks(url)
log.Println(links, err)
```

为返回值命名能够提高可读性：
```go
func Size(rect image.Rectangle) (width, height int)
func Split(path string) (dir, file string)
func HourMinSec(t time.Time) (hour, minute, second int)
```

但不是一定需要命名，例如返回的用于指示操作是否成功的布尔值或者error类型通常都不需要明确的名称，加了反而多余。

### 裸返回

**裸返回（bare return）**是指在一个有着名字的参数列表的函数（如 `a, b, c int`）中，省略其返回值，只写一个`return`的返回，由于参数列表中的实参名以及顺序已经确定，这个裸的`return`确实可以在编译期就转化为完整的返回语句`return a, b, c`。

裸返回最大的作用是减少代码中的重复，但它很少能够起到提高可读性的作用，尤其是对于一些比较长的函数，因此裸返回不应该大量使用。

## 错误处理

将程序中或产生错误、或不产生错误的情况进行分类：
- 一些函数不会产生错误，例如`strings.Contains`或者`strconv.FormatBool`等，这些函数对于任意的输入（前提是能编译）都能够给出结果，而不会出错。其唯一会出错的可能发生在一些非常规的情况，例如内存溢出等，在这种情况下错误的原因与其表现相差甚远（the symptom is far from the cause），并且程序大概率是无法继续运行的。
- 一些函数只要其参数满足要求，就不会发生错误，例如`time.Date`，除非传入的时区是一个`nil`会panic之外，不会产生错误。这个时候它以panic这种形式来递出错误，是为了说明程序中有一个不合理之处导致的bug，它是应该被修复的，而不是保留到最终的程序中。
- 一些函数是否发生错误无法预料，不在程序员的控制范围内（depends on factors beyond the programmer's control），例如一些与I/O相关的函数。这里我觉得作者写的一句话很切中要害，个人翻译如下：*只有幼稚的程序员会认为一个简单的读写操作不会出现问题。而实际上，出现问题的操作越是可靠，我们越有必要知道这是为什么。*这句话说明了即使一个操作很简单，但只要它的错误不在我们的控制范围内，那么我们就不能忽略其可能的错误。

> ... only a naïve programmer believes a simple read or write cannot fail. Indeed, it’s when the most reliable operations fail unexpectedly that we most need to know why. (p127)

在Go中，一个出现错误的函数会返回一个额外的参数，通常作为最后一个参数。如果这个函数出错的原因只有一种，这个参数可以简化为一个布尔类型用来表示成功或失败，例如从字典中取一个值。如果有多重原因，这个参数是一个`error`，它是一个接口类型，为nil时表示没有出错（成功），非nil时总是带有一串字符串信息。

和参数一样，函数的返回值列表一旦确定，其返回的内容就必须时刻遵从其类型和顺序，当函数出错的时候，其作为结果的那个值也会被返回，一般来说这个值应该被忽略，但也有例外，例如用于读取文件的Read函数可能会返回发生错误之前读取的总字节数，这个值有一定的意义并且可以被用于后处理。如果这种部分返回值有意义，应当在文档或注释中明确说明。

Go中的错误处理与很多语言都不一样。在Go中错误处理就是通过返回error来实现的。虽然Go中也有panic这种类似于runtime error/exception的存在，但它是专门用于一些严重问题（而不是程序可以处理的问题），并且往往指示一些需要被修复的bug，例如下标越界、参数不合理等；换句话说，panic是不该存在，但往往因为一些疏忽或不可预料的因素而出现的那类错误。

之所以这么设计，是考虑到在有着exception的编程语言中，一个错误的具体内容（其原因）往往和处理这个错误的控制流（try catch）糅合在一起，导致有的时候一个简单的错误会输出一大长串用户难以理解的调用栈信息（对整个程序结构的描述），而不能提供错误发生的**链路和上下文**。

在Go中错误的处理直接使用`if`语句来进行，这往往使得程序中需要进行错误处理的地方变多，但这也是这个设计的目的之一。

### 五种错误处理模式
#### 1. 错误传播

第一种模式是传播（propagate）错误，这种模式将被调用的函数产生的错误作为调用者函数的错误（或者其错误的一部分）。

一些情况下，我们可以选择直接传播这个错误而不附加上下文。通常这用于一个函数是另一个函数的直接封装的情况，例如将`http.Get`封装为一个带有预设的请求体，专门用于获取特定数据的`fetchXXX`，此时`http.Get`返回的错误就是`fetchXXX`的错误。

一些情况下，我们需要附加上下文。例如当我们在一个获取并解析HTML的函数中获取到了HTML文本，但`html.Parse`返回了错误，此时直接将其错误返回出去并不妥，因为这个函数并不主要是在做parse的工作。我们需要让返回出去的信息具有下列两点信息：
1. 错误发生在解析的过程
2. 解析的对象

这一点可以用`fmt.Errorf`来实现，它在底层调用的是`fmt.Sprintf`，用于从字符串构造一个新的error值，这个字符串里面的描述由我们自行设计和补充。

区分是否需要附加上下文的一个原则是，调用者是否具有被调用的函数所没有的且对错误有意义的信息，如果有，最好将这个信息附加到错误中。

通过这样选择性地附加上下文，在一个很深的调用链路上如果发生了错误，那么这个错误从发生点开始逐步向上传播，每一层都会根据函数的具体语义决定是否附加以及附加怎样的上下文，到最顶层的调用者时，错误的信息中将包含整个链路的上下文，就像NASA的事故调查报告一样：

```
genesis: crashed: no parachute: G-switch failed: bad relay orientation
```

> When the error is ultimately handled by the program’s main function, it should provide **a clear causal chain from the root problem to the overall failure**, reminiscent of a NASA accident investigation ... (p129)

也正因为有这样的分层结构，我们在写英文错误信息的时候不应该使用大写，在写任何语言错误信息的时候都不应该换行。这样，在用`grep`对日志中的错误进行查找时，这个错误信息会自成一行。这样做的好处是明确错误发生的单位：每一次发生的错误唯一对应日志中的一行，其错误原因链由这一整行的内容表示。

在设计错误信息的时候，应该做到谨慎（deliberate）、一致（consistent）。
- 谨慎：错误信息应该包含有意义的描述和有关信息
- 一致：同一个package下可能产生的一系列错误应当有一定的共同点以便统一处理

例如package `os`提供的`Open`、`Read`、`Write`、`Close`在遇到错误的时候不仅会给出其原因，还会给出出错的那个文件的名称，这就满足了一致性；开发者无需在拿到错误以后再去手动加上文件的名称。

#### 2. 重试

对于一些无法预料的问题，我们可以附加重试机制。重试机制通常有下面的两个要素：
- 重试间隔（delay）：重试与重试之间需要有一定的间隔，这个间隔可以是固定的，但也可以是变化的（重试次数的函数），从而实现退避
- 重试终止条件：重试认为虽然发生了错误，但是仍然有拯救的机会，重试不能无限进行下去，需要有一个放弃的条件，通常是重试的次数或者一个给定的超时时间

第一次见到**退避（back-off）**的概念是在计算机网络，它是一种防止网络拥塞的方式。在这里重试时的退避（变化的重试间隔）也有着相同的目的，就是避免固定的时间间隔（类似于轮询）为目标资源带来过于大的压力或者引发连锁反应。退避的delay往往是指数增长的，这样才能发挥退避的作用。通常退避用于网络资源的请求中。

#### 3. 终止执行

如果错误的发生导致程序不能正确地继续执行，调用者可以选择直接终止程序：输出一行日志，然后调用`os.Exit`，这个流程可以用`log.Fatalf`来代替。

#### 4. 屏蔽错误的这部分功能

如果错误的发生只影响程序的一小部分，且这部分对程序不是必需的，调用者可以选择禁用这部分的功能并继续执行。此时需要输出日志来提醒用户程序的功能因为错误发生了一些改变。

#### 5. 彻底忽略

极少数情况下，我们可以直接忽略掉错误，这往往是因为这个错误并不是只有我们在处理。例如当我们使用操作系统中的临时目录来存放临时数据，运行完毕之后去清理这些临时数据，这个清理的函数`os.RemoveAll`，虽然会返回错误，但我们可以忽略，因为操作系统本来就会定期清理临时目录下的数据。

注意前提是临时目录。对于一个确定的非临时目录，这样做不合适。

当我们彻底忽略一个错误的时候，应该在注释中说明，并且尽量使用empty identifier去明确表示这是一个discarding。

### 错误处理的节奏

Go中的错误处理逻辑一般都是扮演着运行条件检查的角色，当发现运行条件不满足（发生了错误），就提前返回，不执行后续的逻辑。这样，错误的处理逻辑一般都放在缩进中（一个if块），并且if块处理完后就返回，这个if不会有else分支；正常流程不会有额外的缩进，使得整个流程清晰。

### `io.EOF`

这是一个特殊的错误，在`io`包下面以`var EOF = errors.New("EOF")`的形式定义。有的时候我们需要根据错误的类型来决定进一步的操作，一个典型的例子是在读取文件的时候。
- 如果我们尝试获取一个文件的完整内容，此时Read返回了错误，代表确实发生了问题。
- 如果我们是一块块地读取文件，每一次只读取一些内容，此时Read返回错误，其中有一种原因是文件已经读取到了末尾，不能读取。这个错误是我们决定停止继续读取的依据。

于是`io`包提供了一个专门的错误用来表示文件不能继续读取，即`io.EOF`。这个错误在分块读取文件时是必须参照的一个停止读取的依据。

```go
in := bufio.NewReader(os.Stdin)
for {
    r, _, err := in.ReadRune()
    if err == io.EOF {
        // 不算错误，而是结束读取的依据
        break
    }
    if err != nil {
        return fmt.Errorf("read failed: %v", err)
    }
    // ...use r...
}
```

**不过，如果是读取整个文件，返回`io.EOF`则表示读取的文件内容不完整。**

## 函数作为值

Go中的函数是一级值（first-class value），可以被赋值给变量、传给函数，并且它们有自己的类型。函数类型的值的零值是nil，其能做的唯一比较也是与nil的比较。调用一个nil function会导致panic。

将函数作为值的语言都可以实现回调函数（callback），不过似乎只有JS中有这种叫法。回调函数让你能够向一个函数的流程中混入自定义的逻辑，例如下列遍历DOM树的函数，通过传入`pre`和`post`两个回调（均选填），可以分别按照先序顺序和后序顺序对整个树进行处理。
```go
func forEachNode(n *html.Node, pre, post func(n *html.Node)) {
    if pre != nil {
        pre(n)
    }
    for c := n.FirstChild; c != nil; c = c.NextSibling {
        forEachNode(c, pre, post)
    }
    if post != nil {
        post(n)
    }
}
```

一个应用是传入如下`pre`和`post`，用来输出一个HTML的树结构。

```go
var depth int
func startElement(n *html.Node) {
    if n.Type == html.ElementNode {
        fmt.Printf("%*s<%s>\n", depth*2, "", n.Data)
        depth++
    }
}
func endElement(n *html.Node) {
    if n.Type == html.ElementNode {
        depth--
        fmt.Printf("%*s</%s>\n", depth*2, "", n.Data)
    }
}
```

这里的动词`%*s`接收两个参数，第一个是其缩进大小，第二个是字符串内容。这里的`depth*2`中，`2`代表的是缩进大小（indent size）。

这两个函数共用一个depth来存储当前遍历到的位置的深度，每深入一层就`depth++`，回溯的时候`depth--`。产生的输出如下
```
<html>
<head>
    <meta>
    </meta>
    <title>
    </title>
    <style>
    </style>
</head>
<body>
    <table>
        <tbody>
            <tr>
                <td>
                    <a>
                        <img>
                        </img>
...
```

## 匿名函数

只有在package level才能定义具名函数（named functions）。而在任何一个表达式中，我们都可以用函数字面量（function literal）去定义一个匿名的函数（anonymous function），这通常用在我们需要传入一个函数，且无需对其复用（at its point of use）的情况下。

匿名的函数具有对其所处的词法环境（lexical environment）的其它变量的访问权限，因此Go中的函数如同JS一样可以形成闭包（closure）。一些人则直接将匿名函数称作闭包。闭包使得函数可以是有状态的。函数能够捕获其所在词法环境中其它变量这一性质，正是函数之间无法比较以及函数属于引用类型的原因。

```go
// squares returns a function that returns
// the next square number each time it is called.
func squares() func() int {
    var x int
    return func() int {
        x++
        return x * x
    }
}

func main() {
    f := squares()
    fmt.Println(f()) // "1"
    fmt.Println(f()) // "4"
    fmt.Println(f()) // "9"
    fmt.Println(f()) // "16"
}
```

下面的代码展示了使用深度优先搜索实现拓扑排序，其中深度优先搜索的部分是使用一个函数字面量来表示的。

```go
func main() {
    for i, course := range topoSort(prereqs) {
        fmt.Printf("%d:\t%s\n", i+1, course)
    }
}

func topoSort(m map[string][]string) []string {
    var order []string
    seen := make(map[string]bool)
    var visitAll func(items []string)
    visitAll = func(items []string) {
        for _, item := range items {
            if !seen[item] {
                seen[item] = true
                visitAll(m[item])
                order = append(order, item)
            }
        }
    }
    var keys []string
    for key := range m {
        keys = append(keys, key)
    }
    sort.Strings(keys)
    visitAll(keys)
    return order
}
```

在这里需要注意的是，如果我们要使用函数字面量去定义一个递归函数，我们必须先将这个函数声明出来（虽然Go中的声明和定义是一体的，这里还是要强调声明），这个函数才能够在函数体内调用自身。其根本原因是在Go中只有package level中标识符的声明顺序不影响调用。另外，如果在顶层使用变量+函数字面量的方式定义递归函数将导致compile error，其原因是标识符不能指向自身。

### 注意：不要直接捕获循环变量

假设我们要创建一系列的目录，做一些操作，然后清除。我们用一个slice来存储每一个目录所对应的那个清除函数：

```go
var rmdirs []func()
for _, d := range tempDirs() {
    dir := d // NOTE: necessary!
    os.MkdirAll(dir, 0755) // creates parent directories too
    rmdirs = append(rmdirs, func() {
        os.RemoveAll(dir)
    })
}
// ...do some work...
for _, rmdir := range rmdirs {
    rmdir() // clean up
}
```

在这里一个很容易形成的误区是省略`dir := d`，在函数中直接使用循环变量`d`。循环变量在这里可以看成是一种引用，对于循环体内部来说，每一次它所看到的都是同一个地址对应的值，这个值在每一次循环时会被更新，所以每一次循环都会看到不同的值。但如果将这个值捕获到函数中用于外部，那么当循环结束以后，这些函数中存储的`d`都指向同一个值，也就是循环结束以后`d`的值。这样，`rmdirs`中的所有函数都只会删除同一个目录。

解决方法是像这样，在内部额外定义一个`dir`，用作`d`的值的拷贝。一般直接将这个变量命名为与循环变量同名，如`d := d`，此处定义的`d`会遮蔽外部的那个`d`。这一行虽然奇怪，但是是必需的。

## 不定参数函数（Variadic Function）

不定参数函数是指参数个数可变的函数。通过给参数列表的最后一个参数的类型前面加上`...`可以构造一个这样的函数。

```go
func sum(vals ...int) int {
    total := 0
    for _, val := range vals {
        total += val
    }
    return total
}
```

在函数内部，这样一个不定长的参数的类型是一个切片。在底层，调用方将先创建一个数组，然后将传入不定长参数的所有值复制到这个数组，然后向函数传入一个指向该数组完整内容的切片。如果已经拿到了要传入不定长参数的那个切片，可以在切片的末尾加上`...`传入参数位置，其含义类似于JS中的spread。

虽然`...int`在内部表现得和`[]int`是一样的，但`func (...int) int`和`func ([]int) int`是两个不同的类型。

## `defer`

一个典型的需要用到`defer`的例子是需要对获取到的资源进行close处理的函数，无论是正常返回还是遇到问题，都需要对资源进行显式close以防泄露。随着函数的逻辑变得越来越复杂，代码中的重复越来越多。我们在这里只是想要表达，当函数返回的时候，一定要释放资源。

`defer`语句在形式上就是一个函数调用前面加上`defer`。它后面的那个表达式会在语句执行的时候被计算，而真正的调用会被推迟（deferred）到当前函数结束。这里的结束不仅包含函数的正常返回，还包括panic。

`defer`语句的数量不限，其函数的调用顺序与`defer`语句的执行顺序相反。

执行`defer`语句最恰当的位置是在成功获取了一个资源之后，即获取资源语句以及错误检查通过的后面。

### 使用`defer`统计函数执行时间

下面的设计利用`defer`对一个函数的执行时间进行简单的统计，这是`defer`一个恰当的应用之一，因为它总是在函数退出的时候调用，符合对一个函数执行时长的测量。

```go
func bigSlowOperation() {
    defer trace("bigSlowOperation")() // don't forget the extra parentheses
    // ...lots of work...
    time.Sleep(10 * time.Second) // simulate slow operation by sleeping
}
func trace(msg string) func() {
    start := time.Now()
    log.Printf("enter %s", msg)
    return func() { log.Printf("exit %s (%s)", msg, time.Since(start)) }
}
```

注意在这里是如何实现使用一个函数就能记录进入和退出两个状态的。`trace`函数做了两件事情：
1. 获取当前的时间，记为`start`，这是进入时的行为
2. 返回一个函数，这个函数捕获了这里的`start`并对其调用`time.Since`，从而给出时间差，这是退出时的行为

将这个函数执行以后，它会立即记录进入的时间，然后返回一个闭包。我们在`defer`语句中对这个闭包进行**调用**，待到函数返回后，这个闭包会真正执行并返回`time.Since`的结果。

### 在`defer`中访问和修改返回值

`defer`的调用时机是在函数返回并且完成了对返回值的赋值之后。

由于匿名函数可以访问它所处词法环境中的其它标识符，如果我们将函数的返回值命名，那么它将可以同时观察到函数的输入和输出。该技巧在一些更为复杂的函数中用作调试很有用。

```go
func double(x int) (result int) {
    defer func() { fmt.Printf("double(%d) = %d\n", x, result) }()
    return x + x
}
_ = double(4)
// Output:
// "double(4) = 8"
```

我们可以甚至可以在`defer`函数里对返回值做修改。
```go
func triple(x int) (result int) {
    defer func() { result += x }()
    return double(x)
}
fmt.Println(triple(4)) // "12"
```

### 在循环中使用`defer`

在循环中使用`defer`，就好像是在安排一连串的任务，等到整个函数返回的时候再真正的执行（且按照循环倒序）。这样做存在一些问题，例如当用于读文件操作的时候，这样做相当于只有在循环结束之后才对文件进行关闭。只要循环在执行，文件资源将一直累积，可能导致资源耗尽（例如file descriptor exhaustion）。更理想的实现应该是打开、处理然后就关掉。

```go
for _, filename := range filenames {
    f, err := os.Open(filename)
    if err != nil {
        return err
    }
    defer f.Close() // NOTE: risky; could run out of file descriptors
    // ...process f...
}
```

因为`defer`是基于其所在的函数起作用的，我们可以将一次文件处理封装到另外一个函数中，在那个函数中使用`defer`就可以实现想要的结果。

```go
for _, filename := range filenames {
    if err := doFile(filename); err != nil {
        return err
    }
}

func doFile(filename string) error {
    f, err := os.Open(filename)
    if err != nil {
        return err
    }
    defer f.Close()
    // ...process f...
}
```

### 有错误返回的函数不适合`defer`

实际开发中对`defer`的使用并不绝对，例如下面的例子。

```go
func fetch(url string) (filename string, n int64, err error) {
    resp, err := http.Get(url)
    if err != nil {
        return "", 0, err
    }
    defer resp.Body.Close()
    local := path.Base(resp.Request.URL.Path)
    if local == "/" {
        local = "index.html"
    }
    f, err := os.Create(local)
    if err != nil {
        return "", 0, err
    }
    // 为什么这里不用defer f.Close()？
    n, err = io.Copy(f, resp.Body)
    // Close file, but prefer error from Copy, if any.
    if closeErr := f.Close(); err == nil {
        err = closeErr
    }
    return local, n, err
}
```

第一个`resp.Body.Close()`用到了`defer`，而`os.Create`返回的`f`在成功获取以后却没有用到`defer`。这是因为`os.Create`等与文件系统有关操作的特殊性。在某些文件系统（如NFS）中，其写入操作（例如这里的创建）发生的错误可能不会马上汇报，而是等到这个文件关闭的时候汇报。`f.Close()`返回的错误可能就是这类错误。

我们不能直接`defer f.Close()`，因为这样就跳过了对这类错误的检查。使用`defer func() { ... }`里面写处理逻辑的方法也不甚妥当，因为在这里我们要做的是将错误向上传播，放在`defer`里面处理错误将导致这个错误与正常的传播流程分离。以上代码中，`io.Copy`执行完毕以后，文件就可以关闭了，此时调用`f.Close()`并获取其中的错误。仅当`io.Copy`没有返回错误（`err == nil`）的时候，我们将错误定为`f.Close()`返回的那个错误进行传播。优先考虑`io.Copy`返回的错误是因为它是最先发生的，最有可能包含根本原因。

## panic

借助Go的静态类型系统，编译器在编译期就可以发现很多错误，使得程序的编译不通过。但有一些问题只有在运行时才能发现，例如下标越界。当这些问题在运行过程中发生时，程序会panic，此时所有执行立即停止，函数的`defer`调用按照相反的顺序，按照调用链自底向上地执行。程序会显示panic日志信息，包括panic值（原因、描述性文本），以及每一个goroutine会有一个对应的调用栈，展示panic的时候被调用的函数及其顺序。这些信息基本足以排查出错误所在。

我们可以借助内置的`panic`函数来手动触发程序panic，这通常用于程序遇到了不可能的情况时。

```go
switch s := suit(drawCard()); s {
case "Spades": // ...
case "Hearts": // ...
case "Diamonds": // ...
case "Clubs": // ...
default:
    panic(fmt.Sprintf("invalid suit %q", s)) // Joker?
}
```

有了手动调用`panic`的能力，我们可以用它来做一些assert的工作，但不能过度，例如去检查一个空指针然后panic。这样做没有必要，是因为运行的时候遇到对空指针的访问本来就会panic，不需要把它写在代码里面。在某些情况下，我们的确需要检查空指针并对外传达信息，此时结合业务逻辑使用error或者panic均可。

```go
func Reset(x *Buffer) {
    if x == nil {
        panic("x is nil") // unnecessary!
    }
    x.elements = nil
}
```

> Although Go’s panic mechanism resembles exceptions in other languages, the situations in which panic is used are quite different. Since a panic causes the program to crash, it is generally used for grave errors, such as a logical inconsistency in the program; diligent programmers consider any crash to be proof of a bug in their code.
> In a robust program, “expected” errors, the kind that arise from incorrect input, misconfiguration, or failing I/O, should be handled gracefully; they are best dealt with using error values. (p149)

虽然panic和其他语言的Exception有着一定的相似之处：都属于运行时的报错，但其目的很不一样。简单来说，panic一般表示预期之外的严重问题，这往往表示程序的设计有缺陷。对于那些预料之内的错误，通常直接使用error处理。

用panic来做assert类的工作的另一个例子，就是编译正则表达式。`regex.Compile`返回一个编译后的正则表达式以及一个error。如果这个正则表达式是由用户提供的，这个error可以算是预期内的错误，我们可以将其返回给用户。如果正则表达式是我们自行编写的、在编译期就确定的值，此时再去检查这一错误就显得有些多余，我们总是希望这个正则表达式能够正确编译，否则程序就不应该运行。这种情况更适合用`regex.MustCompile`，其中Must是Go的一个命名习惯，表示这个函数要么成功，要么导致这个程序退出/panic。`MustCompile`本质上就是`Compile`加上如果发生error就panic的逻辑。

> Readers familiar with exceptions in other languages may be surprised that `runtime.Stack` can print information about functions that seem to have already been “unwound”. Go’s panic mechanism runs the deferred functions before it unwinds the stack.

Go中的`defer`在panic中扮演一个后处理的角色，在`defer`调用执行的时候，调用栈还没有展开，此时可以访问完整的信息。

## recover

当一个程序panic之后，最简单且合理的做法就是直接放弃程序的运行。不过，panic并不一定宣告程序执行的死亡，在一些情况下可以进行恢复（recover）。

恢复的目的可能是让程序能够继续运行，或者处理一下剩下的资源、记录一下日志之类的。例如，在HTTP服务器中，如果某一个路由的handler逻辑里面发生了panic，例如除以0或者下标越界，表示这个handler逻辑写的有问题。但从一些角度来看，这样的错误并不值得让整个HTTP服务器（后端服务）崩溃掉，因为HTTP服务器本身是健康的。既然有500 Internal Server Error这种状态码的存在，为什么不用呢？于是panic+recover就可以被用在这种场景，捕获handler中预料之外的问题，然后自动地关闭链接、向客户端返回500。

`recover`内置函数用于终止一个程序的panic状态，它返回原本panic状态的panic value。考虑到panic机制，`recover`只有在`defer`调用中才有意义，在其它地方调用将没有任何效果并返回nil。panic状态被终止的时候，`recover`所在的那个函数将直接正常返回。由于在`defer`调用中可以直接修改函数的命名返回值，我们可以在`defer`调用中为返回的error中提供panic相关信息。

```go
func Parse(input string) (s *Syntax, err error) {
    defer func() {
        // p != nil 判断是否发生了panic
        if p := recover(); p != nil {
            err = fmt.Errorf("internal error: %v", p) // 改变返回值
        }
    }()
    // ...parser...
}
```

无差别地从panic中恢复（recovering indiscriminately）是一种合理性存疑（dubious）的操作，因为恢复以后的状态往往是无法预测的。此时可能存在未更新完全的数据结构、未完全关闭的文件或者链接以及未解锁的mutex等。如果为了能够将这个错误记录到文件中（需要额外的代码）而强行让程序走一遍recover的流程，很有可能导致一些bug无法及时发现（因为panic本来就说明有bug）。

通常来说，如果recover对应的panic在同一个package中发生，其大概率是合理的。通常情况下我们不应该去recover另一个package中发生的panic。任何用作公共API的函数应该使用error而不是panic来汇报错误。同理，即使是在同一个package中，如果panic的逻辑并不由当前函数负责（比如接收外部传来的回调函数的函数），我们也不应该去recover这个函数导致的panic，因为我们不知道是什么导致了panic。

在前面所说的HTTP服务器的例子中，则恰好违背了上面这段话——不要去处理其它逻辑带来的panic。HTTP服务器这样设计，往往是基于业务的考虑，从逻辑上来讲确实行得通也并不显得不合理，但其代价是可能导致资源的泄露以及程序中其它的问题，这是不可置否的。

综合上面这些考虑，最恰当的recover应当是从“应当recover”的那个panic中recover。为了区分，我们可以在内部定义一个特殊的值用作panic value，recover的时候拿到value后判断是不是这个特殊的值，如果是就可以recover，如果不是，就继续调用panic并传入先前的panic value来延续panic的状态。

```go
func soleTitle(doc *html.Node) (title string, err error) {
    type bailout struct{}
    defer func() {
        switch p := recover(); p {
        case nil:
            // no panic
        case bailout{}:
            // "expected" panic
            err = fmt.Errorf("multiple title elements")
        default:
            panic(p) // unexpected panic; carry on panicking
        }
    }()
    // ...
}
```

但是到这里，既然panic也是expected，为什么不直接用error呢？作者提到，这个例子确实违反了可预期的错误应该直接用error值来表示而不是panic的原则，但该例子集中展示了如何选择性地recover。

实际开发中，将一个error值传来传去可能导致冗余的代码，尤其是在一些深度嵌套的情况。如果为了偷懒/避免代码为了一个error值变复杂，我们确实可以使用这种机制，从某个位置直接触发一个panic并携带一个预定义的panic value，然后就像隔空传话一样将这个错误传到某个作为处理者的defer调用中，让这个defer调用去改变其所在函数的返回值。这样做最大的缺点体现在，panic并不如我们这种用法所假设的那样简单，将其作为一个“隔空传error”的手段是欠考虑的，因为panic会破坏正常的执行流程，导致程序陷入某种可能的中间状态，这会导致bug的产生。
