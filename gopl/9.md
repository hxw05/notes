# Ch9. Concurrency with Shared Variables

本章主要介绍在多个goroutine之间共享同一个变量会带来的问题，对这些问题识别与分析的方法以及解决的常用办法，同时还会介绍goroutine与操作系统中线程的联系。

## 竞态条件

在只有一个goroutine的程序中，语句的执行顺序是由逻辑来决定的。对于有多个goroutine的程序，每个goroutine内部语句的执行顺序亦是如此。但两个不同goroutine中的语句彼此的执行顺序就不能简单得到，可能是A先执行B后执行，可能反过来，也可能是同时执行。

如果我们无法确定两个语句具有先后关系，那么我们就说这两个语句是并发的（concurrent）。

一个并发安全的函数，可以在无需额外同步的情况下，被不同的goroutine中被并发地调用而不出差错。如果一个类型上的所有方法和操作都是并发安全的，那么就说这个类型是并发安全的。

通常情况下，如果一个变量的类型没有被明确注明为并发安全的，我们就不能并发地访问这个变量。为了彻底避免某个变量的并发访问，可以引入互斥锁或者将这个变量限制在某个goroutine的本地作用域内。

但是，通常认为一个在package scope被导出的函数应该是并发安全的，这个函数在访问一些变量的时候必须正确加锁。

### 何为竞态条件

竞态条件是与死锁、活锁、资源饥饿等并列的导致一个函数无法在并发调用下正常运行的原因，其具体表现为多个goroutine调用同一个函数后，由于函数中的操作存在重叠，导致函数的结果不正确。

> Race conditions are **pernicious** because they may remain latent in a program and appear infrequently, perhaps only under heavy load or when using certain compilers, platforms, or architectures. This makes them hard to reproduce and diagnose. (p258)

以下的一个经典的例子展示了竞态条件的问题所在。package bank表示只有一个银行账户的银行，Alice和Bob作为两个操作者向银行中存钱，他们分别对应两个goroutine。
```go
package bank
var balance int
func Deposit(amount int) { balance = balance + amount }
func Balance() int { return balance }
```

```go
go func() {
    bank.Deposit(200) // A1
    fmt.Println("=", bank.Balance()) // A2
}()

go bank.Deposit(100) // B
```

Alice和Bob的操作是并发进行的，也就是说我们无法确定这两个goroutine中语句的具体执行先后顺序。在这种情况下，直觉告诉我们有三种可能性：
- A1,B,A2
    在这种情况下，A2所看到的是300的余额
- A1,A2,B
    在这种情况下，A2看到的是200的余额
- B,A1,A2
    在这种情况下，A2看到的是300的余额

我们发现只有A2操作中的中途读取行为受到了影响。无论以以上三种哪种顺序执行，银行账户最后都是正确的300余额，不会出现金额的损失。但竞态条件的危害恰恰体现在这里，还有第四种情况。第四种情况需要我们从微观的角度来看以上函数中的语句（操作）。

- Balance函数的操作很简单，直接返回balance的值，所以不用太在意
- Deposit函数是一个语句`balance = balance + amount`，这个语句涉及两个步骤：
  - (A1r) 读取balance的值，设为balance1；读取amount的值，设为amount1（amount这一步可忽略）
  - (A1w) 将balance的值更新为`balance1+amount1`

第四种情况是A1r发生在Bob更新值（B完全执行）之前，A1w发生在Bob更新值之后，这会导致Bob更新的值被彻底覆盖掉，因为A1r看到的是旧值，而A1w发生的时候B已经结束。这里就发生了竞态，根本原因是两个goroutine共同访问了同一个balance变量。

这种竞态被称为数据竞态（data race）。数据竞态的发生条件是多个goroutine共同访问同一个变量，且其中的一个访问操作为写操作。

如果涉及到的变量的数据类型大于一个机器字长的大小，那么可能出现更加复杂的情况。下面的代码中，`x[999999] = 1`是一个未定义行为（undefined behaviour）。

简单来看，x有三种可能的值：nil、长度为10的切片、长度为1000000的切片。但切片的底层是一个类似结构体的结构，上面存储有指向底层数组的指针、长度以及容量，在这种位置发生数据竞态，容易这个结构发生混合。例如，x也可能变为长度为1000000，但是底层数组的长度只有10的一个切片。还有很多种其它的可能。因此这是一个未定义的行为。

```go
var x []int
go func() { x = make([]int, 10) }()
go func() { x = make([]int, 1000000) }()
x[999999] = 1 // NOTE: undefined behavior; memory corruption possible!
```

> Even the notion that a concurrent program is an interleaving of several sequential programs is a false intuition. [...] Many programmers—even some very clever ones—will occasionally offer justifications for known data races in their programs: “the cost of mutual exclusion is too high,” “this logic is only for logging,” “I don’t mind if I drop some messages,” and so on. The absence of problems on a given compiler and platform may give them false confidence. *A good rule of thumb is that there is no such thing as a benign data race.* (p260)

benign是良性肿瘤中的良性，上面这段话的意思是，哪怕一些竞态是刻意为之，也最好不要发生，最好就没有竞态。

### 避免数据竞态

根据数据竞态发生的条件：多个goroutine访问同一个变量，且涉及写操作，考虑下面的避免数据竞态的方式。
1. 避免写操作
   
    动态地构建一个map的键值对可以实现数据的懒加载，但这样也导致函数变得并发不安全。为了避免数据竞态，我们直接预构造所有的值，这样将来就只涉及到对这个map的读操作。
2. 避免变量被多个goroutine访问，或者说将变量限制在（confine ... to）一个goroutine中
   
    第八章中爬虫的例子和聊天室的例子，都将一个map封闭在单个goroutine的本地作用域中，从而避免了其它goroutine访问这一变量的可能性。

    而当其它的goroutine确实需要改变这个变量的值时，可以通过通道去通知持有它的goroutine完成更新。这正是Go的圣经“Do not communicate by sharing memory; instead, share memory by communicating”的含义所在。

    这样一个守护着一个变量并且通过通道来接收外部对这个变量的变更请求的goroutine被称为这个变量的监控（monitor）goroutine。

    上面的bank可以改写成monitor goroutine的形式：
    ```go
    package bank

    // Package bank provides a concurrency-safe bank with one account.
    var deposits = make(chan int) // send amount to deposit
    var balances = make(chan int) // receive balance
    func Deposit(amount int)      { deposits <- amount }
    func Balance() int            { return <-balances }
    func teller() {
        for {
            var balance int // balance is confined to teller goroutine
            select {
            case amount := <-deposits:
                balance += amount
            case balances <- balance:
            }
        }
    }
    func init() {
        go teller() // start the monitor goroutine
    }
    ```

    一个变量也不一定总是被限制在同一个goroutine中。不同的goroutine可以通过通道传递地址的方式来传递一个变量。只要这些goroutine在将一个变量的地址通过通道发送给下一个goroutine之后不再访问这个变量，这个变量就是安全的，对这个变量的访问顺序就是顺序的。这种模式称为串行限制（serial confinement）。可以发现这个时候通过通道发送变量很像在移交所有权。
3. 允许多个goroutine访问同一个变量，但是每一次只能有一个goroutine访问。这一点通过互斥锁来实现。
