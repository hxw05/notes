# Ch9. Concurrency with Shared Variables

本章主要介绍在多个goroutine之间共享同一个变量会带来的问题，对这些问题识别与分析的方法以及解决的常用办法，同时还会介绍goroutine与操作系统中线程的联系。

## 竞态条件

在只有一个goroutine的程序中，语句的执行顺序是由逻辑来决定的。对于有多个goroutine的程序，每个goroutine内部语句的执行顺序亦是如此。但两个不同goroutine中的语句彼此的执行顺序就不能简单得到，可能是A先执行B后执行，可能反过来，也可能是同时执行。

如果我们无法确定两个语句具有先后关系，那么我们就说这两个语句是并发的（concurrent）。

一个并发安全的函数，可以在无需额外同步的情况下，被不同的goroutine中被并发地调用而不出差错。如果一个类型上的所有方法和操作都是并发安全的，那么就说这个类型是并发安全的。

通常情况下，如果一个变量的类型没有被明确注明为并发安全的，我们就不能并发地访问这个变量。为了彻底避免某个变量的并发访问，可以引入互斥锁或者将这个变量限制在某个goroutine的本地作用域内。

但是，通常认为一个在package scope被导出的函数应该是并发安全的，这个函数在访问一些变量的时候必须正确加锁。

### 何为竞态条件

竞态条件是与死锁、活锁、资源饥饿等并列的导致一个函数无法在并发调用下正常运行的原因，其具体表现为多个goroutine调用同一个函数后，由于函数中的操作存在重叠，导致函数的结果不正确。

> Race conditions are **pernicious** because they may remain latent in a program and appear infrequently, perhaps only under heavy load or when using certain compilers, platforms, or architectures. This makes them hard to reproduce and diagnose. (p258)

以下的一个经典的例子展示了竞态条件的问题所在。package bank表示只有一个银行账户的银行，Alice和Bob作为两个操作者向银行中存钱，他们分别对应两个goroutine。
```go
package bank
var balance int
func Deposit(amount int) { balance = balance + amount }
func Balance() int { return balance }
```

```go
go func() {
    bank.Deposit(200) // A1
    fmt.Println("=", bank.Balance()) // A2
}()

go bank.Deposit(100) // B
```

Alice和Bob的操作是并发进行的，也就是说我们无法确定这两个goroutine中语句的具体执行先后顺序。在这种情况下，直觉告诉我们有三种可能性：
- A1,B,A2
    在这种情况下，A2所看到的是300的余额
- A1,A2,B
    在这种情况下，A2看到的是200的余额
- B,A1,A2
    在这种情况下，A2看到的是300的余额

我们发现只有A2操作中的中途读取行为受到了影响。无论以以上三种哪种顺序执行，银行账户最后都是正确的300余额，不会出现金额的损失。但竞态条件的危害恰恰体现在这里，还有第四种情况。第四种情况需要我们从微观的角度来看以上函数中的语句（操作）。

- Balance函数的操作很简单，直接返回balance的值，所以不用太在意
- Deposit函数是一个语句`balance = balance + amount`，这个语句涉及两个步骤：
  - (A1r) 读取balance的值，设为balance1；读取amount的值，设为amount1（amount这一步可忽略）
  - (A1w) 将balance的值更新为`balance1+amount1`

第四种情况是A1r发生在Bob更新值（B完全执行）之前，A1w发生在Bob更新值之后，这会导致Bob更新的值被彻底覆盖掉，因为A1r看到的是旧值，而A1w发生的时候B已经结束。这里就发生了竞态，根本原因是两个goroutine共同访问了同一个balance变量。

这种竞态被称为数据竞态（data race）。数据竞态的发生条件是多个goroutine共同访问同一个变量，且其中的一个访问操作为写操作。

如果涉及到的变量的数据类型大于一个机器字长的大小，那么可能出现更加复杂的情况。下面的代码中，`x[999999] = 1`是一个未定义行为（undefined behaviour）。

简单来看，x有三种可能的值：nil、长度为10的切片、长度为1000000的切片。但切片的底层是一个类似结构体的结构，上面存储有指向底层数组的指针、长度以及容量，在这种位置发生数据竞态，容易这个结构发生混合。例如，x也可能变为长度为1000000，但是底层数组的长度只有10的一个切片。还有很多种其它的可能。因此这是一个未定义的行为。

```go
var x []int
go func() { x = make([]int, 10) }()
go func() { x = make([]int, 1000000) }()
x[999999] = 1 // NOTE: undefined behavior; memory corruption possible!
```

> Even the notion that a concurrent program is an interleaving of several sequential programs is a false intuition. [...] Many programmers—even some very clever ones—will occasionally offer justifications for known data races in their programs: “the cost of mutual exclusion is too high,” “this logic is only for logging,” “I don’t mind if I drop some messages,” and so on. The absence of problems on a given compiler and platform may give them false confidence. *A good rule of thumb is that there is no such thing as a benign data race.* (p260)

benign是良性肿瘤中的良性，上面这段话的意思是，哪怕一些竞态是刻意为之，也最好不要发生，最好就没有竞态。

### 避免数据竞态

根据数据竞态发生的条件：多个goroutine访问同一个变量，且涉及写操作，考虑下面的避免数据竞态的方式。
1. 避免写操作
   
    动态地构建一个map的键值对可以实现数据的懒加载，但这样也导致函数变得并发不安全。为了避免数据竞态，我们直接预构造所有的值，这样将来就只涉及到对这个map的读操作。
2. 避免变量被多个goroutine访问，或者说将变量限制在（confine ... to）一个goroutine中
   
    第八章中爬虫的例子和聊天室的例子，都将一个map封闭在单个goroutine的本地作用域中，从而避免了其它goroutine访问这一变量的可能性。

    而当其它的goroutine确实需要改变这个变量的值时，可以通过通道去通知持有它的goroutine完成更新。这正是Go的圣经“Do not communicate by sharing memory; instead, share memory by communicating”的含义所在。

    这样一个守护着一个变量并且通过通道来接收外部对这个变量的变更请求的goroutine被称为这个变量的监控（monitor）goroutine。

    上面的bank可以改写成monitor goroutine的形式：
    ```go
    package bank

    // Package bank provides a concurrency-safe bank with one account.
    var deposits = make(chan int) // send amount to deposit
    var balances = make(chan int) // receive balance
    func Deposit(amount int)      { deposits <- amount }
    func Balance() int            { return <-balances }
    func teller() {
        for {
            var balance int // balance is confined to teller goroutine
            select {
            case amount := <-deposits:
                balance += amount
            case balances <- balance:
            }
        }
    }
    func init() {
        go teller() // start the monitor goroutine
    }
    ```

    一个变量也不一定总是被限制在同一个goroutine中。不同的goroutine可以通过通道传递地址的方式来传递一个变量。只要这些goroutine在将一个变量的地址通过通道发送给下一个goroutine之后不再访问这个变量，这个变量就是安全的，对这个变量的访问顺序就是顺序的。这种模式称为串行限制（serial confinement）。可以发现这个时候通过通道发送变量很像在移交所有权。
3. 允许多个goroutine访问同一个变量，但是每一次只能有一个goroutine访问。这一点通过互斥锁来实现。

## `sync.Mutex`

### 引入

类比在上一章的例子中，为了防止file descriptor耗尽而引入的计数信号灯机制，用于限制能够打开file descriptor的goroutine的最大数量。当我们将计数信号灯的最大数量定为1，也就是至多有一个goroutine能够执行此操作（换句话说，就是没有两个goroutine可以并发地执行此操作）时，它就变为了一个二元信号灯（binary semaphore）。二元信号灯可以用标准库中的类型`sync.Mutex`表示，称为互斥锁。

互斥锁中，获取一个token的操作被称为加锁（lock），反向操作被称为释放锁（unlock），分别由Lock和Unlock函数来完成。由Lock和Unlock所确定的代码片段即为临界区（critical section）。已经拿到锁的goroutine对Unlock函数的调用总是happens-before后面尝试获取锁的goroutine中的流程。

守护（guard）一个变量的锁一般与这个变量的声明写在一起。如果不写在一起，建议加注释。锁、变量以及获取锁对变量进行读写的并发安全函数组合在一起称为一个监控器（monitor），这是一个比monitor goroutine更早提出的概念。

对于较为简单的代码，我们会考虑使用Lock+一段代码+Unlock的组合。但当代码较长，或者带有分支时，为了确保锁能够释放，我们不得不为每一个可能的退出点和异常点都加上Unlock语句：一处获取锁，处处释放锁。这种情况的可维护性和可读性均较差。此时defer语句就很有用，因为它天然地能够保证一个语句在函数退出时能够被调用，因此Lock+`defer Unlock`的组合是最常用的。

这种组合将操作的临界区变为了从Lock语句开始一直持续到函数执行结束，并且defer语句在性能上有一定的损耗，不如直接调用Unlock快。但这点速度并不值得去牺牲可读性（当然，对于简单的代码可以）。既然过早优化是万恶之源，对于复杂的代码，我们只需考虑使用defer。

### 不可重入锁

下面的这个Withdraw函数的实现不正确。Deposit确实加了锁，但下面的“先扣再加回去”逻辑也导致了可能存在某个瞬间（transient）状态，余额是小于零的。这就导致了在这个例子中，Bob试图去买一辆车有可能导致Alice买咖啡提示余额不足（*p264*）。

```go
// NOTE: not atomic!
func Withdraw(amount int) bool {
	Deposit(-amount)
	if Balance() < 0 {
		Deposit(amount)
		return false // insufficient funds
	}
	return true
}
```

一个很容易想到的考虑是，既然这个操作不是原子的（两个Deposit之间可能有插入进来的操作），那么就通过再加一层锁将其变得原子，从而确保两次Deposit一定是按顺序完成的。这样做涉及到了锁的重入（re-entry），这在`sync.Mutex`上不可行，会导致永久的阻塞（在一次Lock后再调用一次Lock）。

要理解这里的不可重入性，需要先明确`sync.Mutex`所保护的不仅仅是临界区的互斥性质，同时还有数据本身的完整性。一个goroutine在获取锁的时候，默认存在这样一个假设：被守护的数据到目前为止是完整的，而不是处于某个中间状态。在临界区内，goroutine可以对这个数据做任何操作，使其处于中间状态，但在释放锁之前，它必须恢复数据的完整性，无论是变为新的状态，还会恢复为其原状态。这样，当数据在递交到下一个临界区时就不会出现假设被打破的问题，相关代码才能够正确地协同。

可重入锁仍然能够保证互斥性质，但会破坏“临界区内的数据一定完整”这一个结论。例如，在一个被锁保护的双向链表中，我们要将一个结点c插入到a和b之间。
```go
mu.Lock()
b := a.next
a.next = c
// ...
c.prev = a
// ...
b.prev = c
mu.Unlock()
```

如果我们在任何一个中间状态去调用针对链表的任意加锁操作，就会破坏临界区内的数据完整性的约束，导致程序的逻辑错误，因为根据假设，我们不会为了去操作一个不完整或处于中间状态的数据而为一个数据加锁，反过来，就是任意加锁的操作的输入一定是完整的数据；只要是加锁的操作，都是对一个完整数据的读或写。

如果确实要进行一些中间的操作怎么办呢？封装成不带锁的函数即可，这些函数不是并发安全的，且只应作为逻辑封装、简化代码的工具使用，而不应该被外部调用。

Go中，尝试重入一个不可重入的锁会导致死锁。

## `sync.RWMutex`

> In a fit of anxiety after seeing his $100 deposit vanish without a trace, Bob writes a program to check his bank balance hundreds of times a second. He runs it at home, at work, and on his phone. The bank notices that the increased traffic is delaying deposits and withdrawals, because all the Balance requests run sequentially, holding the lock exclusively and temporarily preventing other goroutines from running. (p266)

在上面的例子中，`sync.Mutex`用于封装一切临界区，也就是一切访问了被守护的数据的操作，包括读与写。将互斥锁同时用于读与写的保护，在某种程度上忽略了读与写在并发安全条件上的差异以及性能上的差异；频繁的读操作高频率地阻塞了偶尔的写操作，也显得有些不太合理。

多个goroutine同时读取同一个变量，在不加锁的时候也是可以正确进行的。当对变量的操作同时涉及读与写时，这个条件变为只要没有写操作在进行。读与写在互斥意义上可以进行区分，这就引入了多读单写锁（multiple reader, single writer lock），简称读写锁`sync.RWMutex`。

参考以上条件，读写锁对于读操作的要求非常严格，必须为纯粹的读操作。一个简单的getter或者accessor不一定纯粹，因为它可能会导致内部数据的改变，例如某个计数器或者缓存（均为代码层面），这个时候就不能用RLock和RUnlock，而应该退回到Lock和Unlock。如果不确定，应该选择后者。

`sync.RWMutex`仅在下面条件满足时相比`sync.Mutex`有优势：
- 大多数获取锁的goroutine均为读者，即读操作频繁于写操作。这一点也可以从例子中看出来。
- 锁存在争用（contention）情况，即总是会出现多个goroutine尝试获取锁的情形，进而这些goroutine会相互竞争、等待。

以上两点如果不满足，`sync.Mutex`与`sync.RWMutex`在作用上的区分度就会降低，而`sync.RWMutex`的内部机制比前者更加复杂，导致其在与`sync.Mutex`类似的场景下速度更慢，没有优势。

## 内存同步

问题从为什么只读的Balance函数也需要加锁开始。这个函数由于只有一行return语句，所以我们可以认为没有操作可以从中插入，但是从反方向来看，它可能会插入到其它的操作中，例如Withdraw，从而得到不正确的结果。还有一个更为深层的原因是底层的内存同步问题。

现代计算机为了提高效率，会倾向于将写入到内存中的内容先写入到当前核心的缓存中（计算机存储层级），仅当需要的时候再将缓存中的内容刷入到内存中，并且这个刷入的顺序也不是一定的。这就导致了如果两个goroutine不在同一个CPU核心上执行，它们所做的操作就可能暂时对其余goroutine不可见，或者观测到其中一个数据，但看不到另外一个数据。通道和互斥锁这两类同步手段会触发从缓存到内存的刷入，从而让后续流程可以看见先前的所有结果。

考虑下面的代码。

```go
var x, y int
go func() {
	x = 1                   // A1
	fmt.Print("y:", y, " ") // A2
}()
go func() {
	y = 1                   // B1
	fmt.Print("x:", x, " ") // B2
}()
```

该程序中一共有三个goroutine，它们是并发执行的。在单个goroutine中，语句的副作用遵循顺序一致性（sequentially consistent），即后发生的语句可以观测到先发生的语句的结果，这也是为什么除了main goroutine外的两个goroutine都可以直接访问到x,y，因为它们已经被定义。

但不同的goroutine之间则不存在这种机制，因此输出的结果是不确定的；语句A2、B2不一定发生在B1、A1之后，它们可能看到过期（stale）的值。我们可以从排列组合的角度来预测程序的输出。

|1|2|3|4|输出|
|:-:|:-:|:-:|:-:|:-:|
|A1|A2|B1|B2|`y:0 x:1`|
|A1|B1|A2|B2|`y:1 x:1`|
|A1|B1|B2|A2|`x:1 y:1`|
|B1|A1|A2|B2|`y:1 x:1`|
|B1|A1|B2|A2|`x:1 y:1`|
|B1|B2|A1|A2|`x:0 y:1`|

在这里我们刻意考虑了goroutine内部的语句顺序，即A1-A2和B1-B2，但这也是不一定的。原因是两个goroutine内部的语句无关。A2所读取的量不是A1写入的量，B也是一样。一些编译器会根据这一特点交换语句的顺序，换来的结果是可能会出现`x:0 y:0`或者`y:0 x:0`的输出——编译器将Print语句都提前到了goroutine的开头位置。

除了编译器的原因，CPU的缓存到内存的写入延迟也可能会导致双0的输出，因为A和B两个goroutine不一定是在同一个核心上执行的。

以上所有问题都可以通过使用现有的简单一致方法来解决。

## 延迟初始化

对于程序中不一定会使用到的小部分需要初始化的代码，我们可以将其初始化推迟到第一次使用的时候，实现懒初始化（lazy initialization）。在上一章中给出了一个map初始化的例子，使用时，检查map是否为空，如果为空，则调用loadIcons进行初始化。

```go
func loadIcons() {
	icons = map[string]image.Image{
		"spades.png":   loadIcon("spades.png"),
		"hearts.png":   loadIcon("hearts.png"),
		"diamonds.png": loadIcon("diamonds.png"),
		"clubs.png":    loadIcon("clubs.png"),
	}
}

func Icon(name string) image.Image {
    if icons == nil {
        loadIcons() // one-time initialization
    }
    return icons[name]
}
```

Icon函数不是并发安全的，并且其原因不仅限于`icons == nil`可能看到过时的结果而导致多次调用loadIcons。由于缺少同步，loadIcons的代码可能会被重新排列，其中一种可能是：
```go
func loadIcons() {
	icons = make(map[string]image.Image)
	icons["spades.png"] = loadIcon("spades.png")
	icons["hearts.png"] = loadIcon("hearts.png")
	icons["diamonds.png"] = loadIcon("diamonds.png")
	icons["clubs.png"] = loadIcon("clubs.png")
}
```
在这种可能性下面，每一条语句后面都能插入对icons的检测和访问，代表着Icon函数可能访问到不完整的数据，例如尚未赋值完毕的icons。

为了让Icon能够变得并发安全，我们考虑加锁。显然，直接使用`sync.Mutex`可以达到目的，但是会让Icon的效率变得很低。考虑使用`sync.RWMutex`改进，代码如下：

```go{15}
var mu sync.RWMutex // guards icons
var icons map[string]image.Image

// Concurrency-safe.
func Icon(name string) image.Image {
	mu.RLock()
	if icons != nil {
		icon := icons[name]
		mu.RUnlock()
		return icon
	}
	mu.RUnlock()
	// acquire an exclusive lock
	mu.Lock()
	if icons == nil { // NOTE: must recheck for nil
		loadIcons()
	}
	icon := icons[name]
	mu.Unlock()
	return icon
}
```

由于临界区的限制，我们必须将两种可能性分别列举出来。首先使用读锁尝试访问icons，如果可以访问，就直接返回，如果不能访问，就用写锁进行初始化并访问。注意：在获取写锁之前，必须释放读锁；在写锁释放之后、读锁释放之前存在并发插入（interleaving）空间，所以在获取写锁以后还需要做一次判断，否则无法反映最新的状态。

这种写法虽然能够达到目的，并且相较于第一种具有更高的潜在效率，但是代码变得冗长且更加复杂，更容易引入错误（error-prone）。这个时候我们可以借助`sync.Once`来解决这个问题。

`sync.Once`上有一个Do方法，可以确保它自己在整个程序运行过程中只被调用一次。它的内部有一个互斥锁，同时守护一个布尔值和（象征意义上）与之关联的数据结构（例如这里的map）。当Do方法执行完毕后，再次执行它就没有任何效果，甚至不会加锁（因为布尔值会一直保持为已经执行的状态不改变）。

```go
var loadIconsOnce sync.Once
var icons map[string]image.Image

// Concurrency-safe.
func Icon(name string) image.Image {
	loadIconsOnce.Do(loadIcons)
	return icons[name]
}
```

## 实例：并发的函数缓存

最直观的版本如下，这个版本显然不是并发安全的，因为存在对同一个map的并发访问。我们需要围绕这一思想去设计并发安全的函数实现。

```go
// NOTE: not concurrency-safe!
func (memo *Memo) Get(key string) (interface{}, error) {
	res, ok := memo.cache[key]
	if !ok {
		res.value, res.err = memo.f(key)
		memo.cache[key] = res
	}
	return res.value, res.err
}
```

首先可以想到的是引入一个mutex来守护`memo.cache`。这种方法较为激进，属于实现了一个阻塞的缓存（blocking cache），在每次访问之前都需要加锁。这会很大程度地影响性能并且降低缓存带来的收益。

```go{3,9}
// Get is concurrency-safe.
func (memo *Memo) Get(key string) (value interface{}, err error) {
	memo.mu.Lock()
	res, ok := memo.cache[key]
	if !ok {
		res.value, res.err = memo.f(key)
		memo.cache[key] = res
	}
	memo.mu.Unlock()
	return res.value, res.err
}
```

### 思路 1：缩减临界区的范围

第一个思路是将临界区范围缩小，只有在需要访问cache的时候才加锁，这比一整个临界区要快一些。但是由于临界区缩小了，它们之间的区域就有空可插：对于同一个key，`memo.f(key)`可能会被调用多次，因为这个时候缓存可能没有来得及写入，而又有多个并发的`Get(key)`调用。虽然这种情况不影响结果的正确性，却引入了重复的调用。我们更想要的是一种duplicate suppressing（避免重复）的实现。

```go{6}
func (memo *Memo) Get(key string) (value interface{}, err error) {
	memo.mu.Lock()
	res, ok := memo.cache[key]
	memo.mu.Unlock()
	if !ok {
		res.value, res.err = memo.f(key)
		// Between the two critical sections, several goroutines
		// may race to compute f(key) and update the map.
		memo.mu.Lock()
		memo.cache[key] = res
		memo.mu.Unlock()
	}
	return res.value, res.err
}
```

### 思路 2：引入 `ready chan struct {}`

为了避免以上情况，我们需要一个合适的方法来检测数据是否*正在准备*。考虑引入一个通道ready，此时键值对中的值从一个result变为result和ready构成的结构体entry，获取一个键值对中的值的过程存在三种可能。
1. 键不存在，对应数据不存在的情况。看到此情况的goroutine负责准备数据并更新ready
2. 键已经存在，但是正在准备（未准备完毕）。看到此情况的goroutine需要等待准备完毕后，返回缓存的值
3. 键已经存在，且准备完毕，直接返回缓存的值

```go
type entry struct {
    res result
    ready chan struct{}
}
```

更新ready的方法是，负责准备数据的那个goroutine建立entry并创建ready通道（`make(chan struct{})`），准备完毕以后将ready通道关闭，从而*广播*准备完毕的状态。所有在ready通道已经存在但没有关闭的情况下执行的goroutine则使用`<-ready`来使自己阻塞到通道关闭，这一写法在通道关闭之后仍然有意义（不阻塞）。基于上述思路实现的Get如下：

```go
func (memo *Memo) Get(key string) (value interface{}, err error) {
	memo.mu.Lock()
	e := memo.cache[key]
	if e == nil {
		// This is the first request for this key.
		// This goroutine becomes responsible for computing
		// the value and broadcasting the ready condition.
		e = &entry{ready: make(chan struct{})}
		memo.cache[key] = e
		memo.mu.Unlock()
		e.res.value, e.res.err = memo.f(key)
		close(e.ready) // broadcast ready condition
	} else {
		// This is a repeat request for this key.
		memo.mu.Unlock()
		<-e.ready // wait for ready condition
	}
	return e.res.value, e.res.err
}
```

可以看到这里用到的两种同步机制
- mutex用于保护对cache的读写操作，从而保证最先发现`e == nil`的goroutine是唯一的。当发现`e !== nil`时立即解锁。
- 通道用于等待正在准备的数据。这些遇到正在准备数据的goroutine，本质上是在没有同步机制情况下的竞态访问goroutine（也就是duplication）；引入了这种机制以后，它们就可以“乖乖”地等待数据准备完毕后再获取，此时就退化为正常的缓存流程。后续的访问均无需等待，因为`<-e.ready`不再阻塞。

这里的精髓正是引入的这个ready通道，它消除了duplication，实现了一个并发且无重复写入的缓存。

### 思路 3：monitor goroutine 实现

在[避免数据竞态](#避免数据竞态)中第一次提到了monitor goroutine的概念，它是用来避免数据竞态的方法之一。若将一个变量的作用域局限于一个goroutine的内部，使得外部只能通过这个goroutine来对变量进行间接的访问，那么这个goroutine就是这个变量的monitor goroutine。

monitor goroutine的实现有些像一种程序内部微型的C/S架构。因为通道的存在及其设计特点，程序中存在一种“通信”的概念。这样，当外部需要读取缓存内容的时候，就要向缓存的monitor goroutine发请求，并等待其响应。

请求结构体定义如下，其中包含了一个response receive通道，当监控器收到请求后，就可以向其中的response通道进行写入。
```go
// A request is a message requesting that the Func be applied to key.
type request struct {
	key      string
	response chan<- result // the client wants a single result
}
```

监控器goroutine的实现如下。它的启动时机是缓存被创建之时（New）也就是外部可以访问到缓存的同一时刻。server接受一个f参数，用于随后根据情况进行调用，从而获取缓存的内容。cache变量在server内部声明，无法被外部访问，创建之后server便进入一个无限循环，监听发来的请求。在这里不需要任何锁。
- 如果发现缓存不存在，就由它来负责创建entry、准备数据和更新状态。这一点通过`e.call`来实现，并且在一个单独的goroutine中进行，来避免请求的阻塞（否则只能串行）。
- 如果发现缓存已经存在，就由它来发起一个`e.deliver` goroutine，用于监听准备的情况并返回数据。

这里所使用的goroutine比前面的方法要多，是这种实现的一个特点。由于彼此之间的通讯靠的是通道这种结构，一个goroutine可以根据需要，将它收到的通道转交给另一个goroutine来处理。唯一需要的注意的是每一个goroutine都需要有明确的职责且避免阻塞（泄露）。

```go
func (memo *Memo) server(f Func) {
	cache := make(map[string]*entry)
	for req := range memo.requests {
		e := cache[req.key]
		if e == nil {
			// This is the first request for this key.
			e = &entry{ready: make(chan struct{})}
			cache[req.key] = e
			go e.call(f, req.key) // call f(key)
		}
		go e.deliver(req.response)
	}
}
func (e *entry) call(f Func, key string) {
	// Evaluate the function.
	e.res.value, e.res.err = f(key)
	// Broadcast the ready condition.
	close(e.ready)
}
func (e *entry) deliver(response chan<- result) {
	// Wait for the ready condition.
	<-e.ready
	// Send the result to the client.
	response <- e.res
}
```

> It's possible to build many concurrent structures using either of the two approaches—shared variables and locks, or communicating sequential processes—without excessive complexity.
> 
> It's not always obvious which approach is preferable in a given situation, but it's worth knowing how they correspond. Sometimes switching from one approach to the other can make your code simpler. (p279)
