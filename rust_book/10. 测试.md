# 10. 测试

“测试”在我过去接触的一些编程语言中从未作为一种语言本身的特性来介绍，Rust Book 中的这一章为我这种没有接触过系统性测试的人提供了一些基础的参考信息。

## 编写测试

Rust 中，借助 `test` 注解可以让一个函数变为测试项。运行 cargo test 后，这些测试项将被运行，并给出相关的测试结果：如果没有 panic，就代表测试通过。

下面代码中的 `#[cfg(test)]` 注解表示该模块只会在 cargo test 的时候被编译

```rust
pub fn add(left: u64, right: u64) -> u64 {
    left + right
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn exploration() {
        let result = add(2, 2);
        assert_eq!(result, 4);
    }

    #[test]
    fn another() {
        panic!("Make this test fail");
    }
}
```
*exploration 和 another 函数都被标记为测试项，在 cargo test 命令执行时会被分别运行*


### `assert!` 宏

assert! 宏类似于其它语言中存在的 assert 关键字，它的第一个参数接受一个布尔值，如果该布尔值为 false，就会触发一个 panic。第二个参数是一个可选的自定义报错字符串。

assert_eq! 宏和 assert_ne! 宏类似于向 assert 传递 `==` 和 `!=` 表达式，第一、二个参数分别为左、右操作数。

当 eq 或 ne 断言失败时，它们会使用**调试格式**打印出其参数，这意味着被比较的值必须实现了 PartialEq 和 Debug trait。这是因为对于自定义的结构体和枚举，需要实现 PartialEq 才能断言它们的值是否相等；需要实现 Debug 才能在断言失败时打印它们的值。

### should_panic

:::warning
`#[should_panic]` 应当在 `#[test]` 之后
:::

should_panic 是一个注解，被它注解的测试必须触发 panic 才算通过。该注解接受一个 expected 配置项，用来自定义测试不通过提示内容。

```rust{6}
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    #[should_panic(expected = "less than or equal to 100")]
    fn greater_than_100() {
        Guess::new(200);
    }
}
```

### Result 作为测试结果

测试函数（标注有 `#[test]` 的函数）可以返回一个 Result，为 Ok 时代表测试通过，失败时以 Err 中的值来作为失败信息。

注意：返回 Result 的测试不应使用 should_panic 注解。

## 控制测试的运行

cargo test 的默认行为是**并发**运行所有的测试，并且屏蔽掉测试过程中的**程序输出**。

因此，如果不指定任何参数进行测试，必须确保每个测试之间是独立的，否则会因为并发执行而出现问题。

为了控制测试的行为，可以为其附带参数。这些参数与传递给程序本身的参数一样，需要与传递给 cargo 本身的参数用 `--` 分隔开。

```bash
cargo test abc -- --test-threads=1
```

cargo test 后跟的第一个参数，表示要运行的目标函数的*路径*的子串。例如，`cargo test add` 会运行 `tests::add_three_and_two` 和 `tests::add_two_and_two` 两个测试函数，因它们的路径中都包含了 add。路径中的“tests”也可以被匹配到。

可用的一些 Flag:
- `--test-threads=?` 调整运行线程数量
- `--show-output` 展示程序本身运行的输出
- `--ignore` 执行所有被标有 ignore 的测试：如果在 `#[test]` 后跟 `#[ignore]`，则该测试默认不会执行。只有在明确指定函数名，或指定相关 flag 才会执行。
- `--include-ignore` 执行所有测试，包括标有 ignore 的测试
- `--test [Crate 名]` 运行特定 crate 中的所有测试

## 测试的组织结构

### 单元测试 Unit Test

> 单元测试的目的是在与其他部分隔离的环境中测试每一个单元的代码，以便于快速而准确地验证某个单元的代码功能是否符合预期。

#### cfg(test) 注解

`#[cfg(test)]` 是针对**模块**的注解，表示只有在运行测试的时候才编译和运行这些测试代码，在 cargo build 时则完全忽略。

使用该注解的一大原因，是**单元测试**往往与相关的源代码放在一起（例如同一个文件里紧挨着），因此可能会被一起编译。这与**集成测试**不同——集成测试往往与源代码是隔离的（单开目录），因此一般不需要特别注解。

:::tip
`#[cfg(...)]` 本身的含义是，仅当给定特定配置项（这里是test），才包含接下来的代码。
:::

### 集成测试 Integration Test

集成测试是单元测试的拓展，通常是为了测试库的多个部分是否可以一起正常工作（从外部调用）。集成测试通常放在单独的 tests 目录内，与 src 同级。

在 tests 中的每一个文件都会被视为一个独立的 (binary) crate 进行编译，且仅当 cargo test 时才会被编译。

集成测试总是在单元测试之后运行，如果单元测试失败，则集成测试不会运行。

#### 在集成测试中组织复用代码

:::info
此部分内容需要在实际编写代码的时候才有参考价值。
:::

tests 下的所有文件都会被视为一个 crate 独立编译，无论其内部是否真的含有测试。为了避免一些仅供复用而非测试的代码被视为测试而编译，需要以传统的（旧的）模块命名方式来进行组织：创建一个文件夹，然后再在内创建一个 `mod.rs`。

#### 关于 binary crate

:::info
此部分内容需要在实际编写代码的时候才有参考价值。
:::

作为 binary crate root 的 main.rs 不能被 use，也就不能被测试，最好的方法是创建一个 main.rs 和一个 lib.rs，将需要测试的逻辑放在 lib.rs 内，针对 lib.rs 进行测试。

从逻辑上来说，如果这样的测试无误，那么 main.rs 的运行也应当没有问题，也就不需要对 main.rs 进行测试了。
